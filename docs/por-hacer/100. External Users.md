> **Estado:** 🟡 Diseño acordado — pendiente de implementación

> **Prioridad:** Alta (acceso a registro de palets en almacén externo)

> **Módulo de maquila completo:** 🔴 Fuera de scope por ahora — ver sección "Trabajo Futuro"

---

# Contexto y motivación

Determinados proveedores externos (maquiladores, almacenistas) necesitan interactuar con PesquerApp para registrar operativa que ocurre en sus instalaciones pero que pertenece al inventario del tenant. El caso de uso inmediato es el **maquilador**: recibe materia prima del tenant, la procesa, y debe registrar los palets de producto terminado directamente en la app.

Esta funcionalidad fue parcialmente implementada en el pasado (acceso restringido al StoresManager de un almacén vinculado) pero se deprecó por falta de estructura genérica. Este documento establece el diseño correcto y escalable.

---

# Decisiones de diseño acordadas

## 1. Los usuarios externos viven en el schema del tenant

No existe una entidad global en la BD central. Cada tenant gestiona sus propios usuarios externos de forma completamente independiente. Si un mismo maquilador trabaja con dos tenants distintos, tendrá dos accesos diferentes — esto es intencionado y correcto: **ningún tenant debe saber si otro usa el mismo proveedor**.

## 5. Gestión de external\_users desde el panel del tenant

El tenant gestiona sus usuarios externos desde un **apartado propio en el panel de administración** — con su tabla de entidades y acciones CRUD, igual que el resto de entidades simples del sistema (clientes, proveedores, etc.). Desde ese apartado el tenant puede crear, editar, activar/desactivar usuarios externos y reenviar manualmente el magic link de acceso.

## 2. Autenticación via Magic Link / OTP

Igual que los usuarios internos. Sin contraseña persistente. El acceso se realiza mediante el flujo ya existente de **magic link + OTP** sobre el modelo de tokens de Sanctum:

- El `ExternalUser` recibe un email con enlace y código OTP.
- Al verificar el enlace/código se genera un token Sanctum (`personal_access_tokens`), igual que para un `User` interno.
- Mientras el token siga activo, la sesión **no tiene expiración corta** — persiste hasta logout explícito o revocación del token, ya que el usuario externo puede entrar múltiples veces al día y no puede estar solicitando nuevos códigos constantemente.

El magic link solo se usa para el primer acceso o cambio de dispositivo.

## 3. Actor externo y guard de autenticación

Los usuarios externos **no usan el guard `web` de sesión**, sino el mismo mecanismo de tokens Sanctum que los usuarios internos (guard `sanctum` en las rutas API v2), pero con:

- Un **modelo distinto** (`ExternalUser`) viviendo en la BD de tenant.
- Un **conjunto de rutas y middleware específico** para externos (sin dependencia del middleware de roles interno).

Esto permite que las Policies y servicios distingan el tipo de actor (interno vs externo) y apliquen restricciones diferentes sin mezclar lógica de negocio.

## 4. Almacén externo como tipo del almacén existente (`Store`)

No se crea una entidad nueva `external_warehouse`. Se añade un **tipo** a la entidad `Store` existente (tabla `stores`) y una referencia al usuario externo responsable. Así se reutiliza toda la lógica de almacenes (palets, cajas, movimientos, trazabilidad) sin duplicar infraestructura.

# Modelo de datos

## Tabla `external_users` (en el schema del tenant)

```
external_users
  id
  name
  company_name
  email
  type: enum(maquilador, almacenista_externo, transportista, lab_calidad, ...)
  is_active
  notes
  created_at / updated_at
```

## Cambios en `stores`

```
stores
  ...
  store_type: enum(interno, externo)      ← campo nuevo
  external_user_id: FK → external_users   ← nullable, solo si tipo externo
```

> **Restricción**: un almacén de tipo `externo` solo puede estar vinculado a un único `external_user`. No se permite compartir almacenes entre varios usuarios externos. Un mismo `external_user` sí puede tener varios almacenes vinculados a él.

## Gestión de sesiones de usuarios externos

No se introduce una tabla de sesiones específica para usuarios externos. Las sesiones se gestionan igual que para usuarios internos:

- Tokens de acceso en la tabla estándar `personal_access_tokens` de Sanctum.
- Tokens de magic link / OTP en la tabla `magic_link_tokens` por tenant.

Opcionalmente, en el futuro se podría añadir metadata adicional (ej. `last_used_at`) o auditoría de actividad específica en una tabla complementaria, pero no forma parte del alcance inicial.

# Flujo operativo (caso: maquilador)

1. El tenant crea un `external_user` de tipo `maquilador` con nombre y email del proveedor, desde el apartado de Usuarios Externos del panel admin
2. El tenant crea (o configura) un `store` de tipo `externo` vinculado a ese `external_user`
3. **El sistema envía automáticamente un magic link** al email del usuario externo al crearlo. El tenant puede reenviar el magic link manualmente desde el panel en cualquier momento
4. El usuario externo accede intentando logarse con su email por el **mismo login que los usuarios internos** — el sistema detecta que es `external_user` del tenant (resuelto por subdominio) y le envía el magic link / OTP
5. El frontend detecta el tipo `external_user` en la respuesta de autenticación y renderiza únicamente el StoresManager — sin menú admin, sin otros módulos, sin opciones de pedidos
6. El StoresManager solo muestra los almacenes vinculados a ese `external_user`
7. Puede **crear y editar palets** dentro de esos almacenes, incluyendo gestión de cajas y productos
8. El tenant ve esos palets en su inventario igual que cualquier almacén interno
9. Toda operación queda registrada en el historial del palet (campo JSON ya implementado en el sistema) — la trazabilidad persiste aunque el `external_user` sea desactivado o eliminado posteriormente

# Consideraciones del StoresManager para external\_user

El `external_user` accede al mismo StoresManager que los usuarios internos, pero con restricciones en puntos concretos:

**Vista de almacenes** — el listado inicial solo muestra los almacenes vinculados al `external_user`. El filtrado ocurre en el backend: el endpoint de almacenes devuelve únicamente los suyos cuando el actor autenticado es un `ExternalUser`.

**Crear palet** — no requiere restricción especial. El StoresManager asume el contexto del almacén desde el que se pulsa crear, por lo que si el `external_user` solo ve sus almacenes, inevitablemente solo puede crear palets en ellos.

**Detalle de palet** — el `external_user` puede ver el detalle completo del palet, incluyendo productos, lotes, fechas y trazabilidad. También puede ver y gestionar las cajas y productos asociados al palet, ya que los necesita para registrar correctamente la producción. Lo que no puede ver son opciones vinculadas a pedidos — esas opciones no deben aparecer en su interfaz.

**Traspaso de palet** — punto crítico. El selector de almacén destino debe mostrar únicamente los almacenes vinculados al `external_user`, no todos los del tenant. El backend valida además en la Policy que tanto origen como destino le pertenecen antes de ejecutar el movimiento.

**Histórico / trazabilidad** — si existe alguna vista de movimientos, solo muestra los de sus almacenes.

**Eliminar palet** — no está permitido. El `external_user` puede crear y editar palets pero no eliminarlos.

**Usuario desactivado** — si el tenant desactiva un `external_user`, se le niega el acceso con mensaje informativo y se invalidan todas sus sesiones activas de inmediato.

Regla general: **el filtrado es siempre responsabilidad del backend**. La API nunca devuelve datos de almacenes ajenos a un `ExternalUser` autenticado, independientemente de lo que solicite el frontend. El frontend simplemente renderiza lo que recibe.

# Cambios en la entidad Store

## Migración de BD

Dos campos nuevos en la tabla `stores` del schema de cada tenant:

* `store_type` — `enum('interno', 'externo')`, valor por defecto `interno`. Todos los almacenes existentes quedan como `interno` al migrar.
* `external_user_id` — FK nullable a `external_users.id`. Solo relevante cuando `store_type = 'externo'`. Con `ON DELETE SET NULL` para no perder el almacén si el usuario externo se elimina.

## Validaciones en backend (creación y edición de almacén)

* Si `store_type = 'externo'`, el campo `external_user_id` es **obligatorio**.
* Si `store_type = 'interno'`, el campo `external_user_id` debe ser **nulo**.
* Un mismo `external_user_id` puede vincularse a varios almacenes (el maquilador puede tener más de un almacén).
* No se permite asignar el mismo almacén a más de un `external_user`.

## Formulario de creación / edición de almacén (frontend)

* Se añade un campo `Tipo de almacén` con selector `interno` / `externo`.
* Cuando se selecciona `externo`, aparece un selector de `Usuario externo` con los `external_users` activos del tenant.
* Cuando se selecciona `interno`, el selector de usuario externo desaparece y se limpia el valor.
* En edición, si el almacén ya tiene palets, mostrar aviso informativo al cambiar el tipo (no bloquear, solo advertir).

## Listado de almacenes — tabla de entidad genérica (frontend)

Nuevas columnas visibles en la tabla:

* `Tipo` — badge visual `Interno` / `Externo`.
* `Usuario externo` — nombre del `external_user` vinculado, o vacío si es interno.

Nuevos filtros en el listado:

* **Filtro por tipo** — selector `Todos` / `Interno` / `Externo`.
* **Filtro por usuario externo** — selector con los `external_users` del tenant, visible solo cuando el filtro de tipo es `Externo` o `Todos`.

# Impacto técnico

* **Nuevo modelo** `ExternalUser` en el schema de tenant (con `UsesTenantConnection` y soporte de tokens Sanctum si se expone por API)
* **Nuevos campos** `store_type` + `external_user_id` en tabla `stores`
* **Extensión de Policies / servicios** de `Store` y `Pallet` (o nuevas policies específicas) para aceptar `ExternalUser` como actor con scope restringido:
  * Usuarios internos (`User`) mantienen el comportamiento actual (ven todos los almacenes del tenant).
  * Usuarios externos (`ExternalUser`) solo pueden ver y operar sobre sus `stores` y palets asociados.
* **Rutas de autenticación / acceso específicas para externos** reutilizando el flujo de magic link/OTP existente, pero sin pasar por el middleware de roles interno.
* **Sin portal separado en frontend** — el usuario externo accede por el mismo login que los usuarios internos. Tras autenticarse, el sistema detecta que es `ExternalUser` y renderiza únicamente el StoresManager filtrado a sus almacenes vinculados. No hay ruta ni layout diferente.
* **Reutilización de** `MagicLinkService` adaptado al modelo `ExternalUser`

# Seeders de desarrollo

Deben crearse seeders realistas que se integren con los seeders existentes (`DatabaseSeeder.php`). Se añaden dos nuevos seeders al final de la cadena de llamadas, después de los seeders de almacenes (`StoreSeeder`):

```php
// En DatabaseSeeder.php — añadir al array $this->call([...]):
ExternalUsersSeeder::class,       // después de los seeders de stores
ExternalStoresSeeder::class,      // depende de ExternalUsersSeeder
```

## ExternalStoresSeeder

Crea almacenes (`stores`) de tipo `externo` vinculados a los usuarios externos del seeder anterior, y asegura que los almacenes existentes (creados por el seeder de stores previo) conserven `store_type = 'interno'`:

```php
<?php

namespace Database\\Seeders;

use Illuminate\\Database\\Seeder;
use App\\Models\\Store;
use App\\Models\\ExternalUser;

class ExternalStoresSeeder extends Seeder
{
    public function run(): void
    {
        $figueiras = ExternalUser::where('email', 'antonio@congeladosfigueiras.es')->first();
        $ferreira  = ExternalUser::where('email', 'joao.ferreira@frigorificoatlantico.pt')->first();

        // Almacén del maquilador — puede tener varios
        Store::create([
            'name'             => 'Cámara Figueiras — Línea Pulpo Cocido',
            'store_type'       => 'externo',
            'external_user_id' => $figueiras->id,
            'description'      => 'Cámara de producto procesado en instalaciones de Congelados Figueiras. Almacena palets de pulpo cocido congelado pendientes de recogida.',
            'is_active'        => true,
        ]);

        Store::create([
            'name'             => 'Cámara Figueiras — Línea Pulpo Crudo',
            'store_type'       => 'externo',
            'external_user_id' => $figueiras->id,
            'description'      => 'Segunda cámara en instalaciones de Figueiras para pulpo crudo en proceso.',
            'is_active'        => true,
        ]);

        // Almacén del almacenista externo
        Store::create([
            'name'             => 'Frigorífico Atlântico — Nave 3',
            'store_type'       => 'externo',
            'external_user_id' => $ferreira->id,
            'description'      => 'Nave frigorífica en Viana do Castelo. Stock de producto terminado en espera de expedición a clientes europeos.',
            'is_active'        => true,
        ]);
    }
}
```

> **Nota**: los seeders de almacenes internos existentes no necesitan modificarse — al no especificar `store_type`, el valor por defecto `interno` que establece la migración los cubre automáticamente.

# Trabajo futuro — Módulo completo de Maquila

> ⚠️ Este módulo es sustancialmente más complejo y queda fuera de scope por ahora. Se documenta aquí como referencia para cuando se aborde.

El módulo completo de maquila implica un flujo bidireccional entre tenant y maquilador con entidades propias, similar en complejidad a un pedido:

* **Entidad `MaquilaProcess`** — equivalente a un pedido pero para envío a maquilar. Registra qué materia prima sale del tenant, hacia qué maquilador, con qué especificaciones
* **Flujo de estados** — borrador → enviado → en proceso → completado → recibido
* **Acciones del tenant** — crear proceso, enviar materia prima, confirmar recepción de producto terminado
* **Acciones del maquilador** — confirmar recepción de MP, registrar producción (palets de producto terminado asociados al proceso), notificar al tenant
* **Integración con producción** — los palets registrados por el maquilador deben integrarse con el módulo de producción y consumo de cajas/palets existente
* **Documentación asociada** — albaranes de entrada/salida, documentos sanitarios si aplica

Esta entidad requiere análisis específico de cómo encaja en la implementación actual de `Production`, `Stock` y el flujo `Box → Pallet`.
